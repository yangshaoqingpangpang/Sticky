# 微信提醒 · 适配账号申请与部署完整指南

> 面向「侧边待办」微信模板消息提醒功能。读者：你（项目负责人）。
> 后端代码：`Sticky/wechat-notify-server/`（Node.js + TypeScript，独立系统）。
> 本文从零讲清：申请什么账号 → 怎么配 → ID 填到哪 → 怎么部署 → 怎么联调。

---

## 0. 全局架构（先有个图）

```
候选人微信  ──扫码关注──►  微信服务器  ──回调(关注事件)──►  你的后端(/wechat)  ──绑定 openid
                                                              │
macOS app  ──REST(/api/*)──►  你的后端  ──存提醒──►  定时扫描到点  ──模板消息API──►  微信  ──►  候选人收到提醒
```

三样东西缺一不可：
1. **认证服务号**（不是订阅号、不是个人号）——才有「模板消息 + 带参二维码 + 网页回调」能力。
2. **已备案的 https 域名**——微信回调和接口强制 https，国内服务器域名必须 ICP 备案。
3. **一台公网服务器**（你选了 server 47）——跑本后端 + nginx 反代。

> ⚠️ 你目前三样都还没有。下面按顺序办，**办一项就把对应 ID 填进 `.env`**。

---

## 1. 申请认证服务号

1. 打开 <https://mp.weixin.qq.com> → 「立即注册」→ 选 **订阅号/服务号** 里的 **服务号**（务必选服务号，订阅号没有模板消息权限）。
2. 主体建议选 **企业/个体工商户**（个人主体的服务号无法微信认证，也就拿不到模板消息）。准备：营业执照、对公账户或法人信息。
3. 注册完成后：公众平台 → 「微信认证」→ 提交认证（一年一审，约 300 元认证费）。**认证通过后**才会出现「模板消息」「自定义菜单」「网页授权」等高级接口。
4. 记下 **AppID** 与 **AppSecret**：公众平台 →「设置与开发 → 基本配置」。
   - AppSecret 只显示一次，丢了要重置。重置后旧的立即失效。

> 没有企业主体？两条路：① 借/注册一个个体工商户做主体；② 改用「微信小程序订阅消息」方案（见附录 B），小程序个人主体也能做，但每条提醒需用户当次授权，体验弱一些。

---

## 2. 域名备案 + https

微信要求回调与接口走 https，且域名指向的服务器在国内时必须 **ICP 备案**。

1. 在阿里云（你 server 47 在阿里云）控制台为域名做 **ICP 备案**（约 7–20 个工作日，需主体资料）。
2. 备案通过后，把一个子域名解析到 server 47 公网 IP，例如 `wx.你的域名.com`。
3. 申请该子域名的 **免费 SSL 证书**（阿里云数字证书服务），下载 nginx 格式证书。
4. 本功能建议用独立子域名（如 `wx.xxx.com`），不要和现有项目域名混。

> 阿里云安全组只放行 22/80/443/18088。微信回调走 **443(https)**，由 nginx 终结后反代到本后端内网端口 **9528**。

---

## 3. 公众平台：服务器配置（回调）

公众平台 →「设置与开发 → 基本配置 → 服务器配置」：

| 字段 | 填什么 |
|---|---|
| URL | `https://wx.你的域名.com/wechat` |
| Token | 自定义一串字母数字（如 `sticky_wx_8f3a`），**和后端 `.env` 的 `WECHAT_TOKEN` 必须一模一样** |
| EncodingAESKey | 点「随机生成」；本后端用明文模式，可不依赖它，但消息加密模式选「明文模式」最省事 |
| 消息加解密方式 | 选 **明文模式**（本后端按明文实现） |

点「提交」——微信会立刻 GET 你的 `/wechat?signature=...&echostr=...`，后端校验通过并回显 echostr 才算保存成功。**所以保存前后端必须已部署且 Token 已配好**（见第 6 步顺序）。

同页面「IP 白名单」：把 server 47 的公网出口 IP 加进去，否则调 access_token 会被拒。

---

## 4. 申请模板消息，拿模板 ID

1. 公众平台 →「广告与服务」或「功能 → 模板消息」（认证服务号才有此入口）→ 开通模板消息。
2. 从模板库添加，或申请一个「任务提醒 / 待办提醒」类模板。本后端默认用三个字段：

   | 字段名(`.data` key) | 含义 | 示例 |
   |---|---|---|
   | `thing` | 待办内容 | 周五前出多端同步接口文档 |
   | `time` | 提醒时间 | 2026-06-26 18:00 |
   | `remark` | 备注 | 来自侧边待办 |

   申请模板时，模板正文写成类似：
   ```
   {{thing.DATA}}
   时间：{{time.DATA}}
   {{remark.DATA}}
   ```
   > 若你拿到的模板字段名不同（如 `keyword1/keyword2`），把 `src/scheduler.ts` 里的 `data` 字段名改成模板实际字段即可（只改那几行）。
3. 拿到 **模板 ID**（一长串），填进 `.env` 的 `WECHAT_TEMPLATE_ID`。

---

## 5. 把所有 ID 填进后端配置

在 `wechat-notify-server/` 下 `cp .env.example .env`，填：

```
PORT=9528
WECHAT_APPID=wx 开头的 AppID
WECHAT_APPSECRET=基本配置里的 AppSecret
WECHAT_TOKEN=与公众平台服务器配置里完全一致的 Token
WECHAT_TEMPLATE_ID=第4步拿到的模板ID
API_KEY=给 app 用的随机长串(app 端也要填一样的)
DATA_FILE=./data/store.json
DETAIL_BASE_URL=               # 可留空(模板消息点击跳转用)
```

`.env` 已 gitignore，**切勿提交**。

## 6. 部署到 server 47（正确顺序很重要）

> 微信「保存服务器配置」时会立即回调你的后端，所以**必须先把后端跑起来 + Token 配好，再去公众平台点保存**。

```bash
# 6.1 本地构建产物上传(或服务器上 git pull 后构建)
#     先确认 server 47 已装 Node ≥18

# 6.2 在 server 47 上:
cd /opt/wechat-notify-server          # 自定目录
# 放好代码 + .env
npm ci
npm run build
# 用 pm2 常驻
npm i -g pm2
pm2 start dist/index.js --name wechat-notify
pm2 save && pm2 startup            # 开机自启

# 6.3 nginx 反代(新增 server 块, 走 443)
```

nginx 片段（`/etc/nginx/conf.d/wx.conf`）：
```nginx
server {
    listen 443 ssl;
    server_name wx.你的域名.com;
    ssl_certificate     /etc/nginx/cert/wx.pem;
    ssl_certificate_key /etc/nginx/cert/wx.key;

    location / {
        proxy_pass http://127.0.0.1:9528;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```
```bash
nginx -t && nginx -s reload         # 改完必须 -t 再 reload
```

```bash
# 6.4 验证后端可达(公网)
curl https://wx.你的域名.com/health   # 期望 {"ok":true,"wechatConfigured":true}

# 6.5 回公众平台「服务器配置」点【提交】——此刻才会校验通过
```

> 部署后按 `~/.claude/rules/deploy-port-registry.md` 把 9528 端口、wx 子域名、pm2 进程名登记到 `~/Desktop/本机部署.md`，避免与其他项目冲突。

## 7. 联调自检（按顺序勾）

- [ ] `curl https://wx.../health` → `wechatConfigured:true`
- [ ] 公众平台服务器配置「提交」成功（说明回调签名 OK）
- [ ] app（或 curl）调 `POST /api/candidates/bind-qr` → 返回 `qrUrl`
- [ ] 用候选人微信扫该 `qrUrl` 图 → 关注服务号 → `GET /api/candidates` 看到该人 `openid` 已填、`boundAt` 有值
- [ ] `POST /api/reminders`（remindAt 设成 1 分钟后）→ 等待 → 候选人微信收到模板消息
- [ ] `GET /api/reminders/:id` → status 变 `sent`

任一步失败的排查：
- 回调保存失败 → Token 两边不一致 / 后端没起 / 443 不通。
- access_token 报错 40164 → IP 白名单没加 server 47 出口 IP。
- 模板发送 40037/47003 → 模板 ID 错 / 模板字段名和 `scheduler.ts` 的 `data` 对不上。
- 扫码不绑定 → 看后端日志有没有收到 `/wechat` POST；scene_str 是否回传。

## 8. macOS app 侧（第二阶段，后端上线后做）

app 需要：
1. 设置里填 **后端地址**（`https://wx.你的域名.com`）+ **API_KEY**。
2. 候选人管理页：调 `bind-qr` 拿 `qrUrl`，用 `AsyncImage`/`NSImage` 展示二维码给对方扫；列表展示绑定状态。
3. 待办上「微信提醒 @候选人 + 时机」开关 → 到点本地算好 `remindAt` 调 `POST /api/reminders`。
4. 失败兜底：后端不可达或候选人未绑定 → 回退本地 Mac 提醒并标注「微信未送达」。

> 这部分等后端有正式 URL 后再实现联调（无法在后端就绪前端到端验证）。

## 附录 A：苹果审核合规（二维码）

二维码本身合规。把握：① 微信提醒是**可选附加功能**，核心待办不依赖它；② 不通过扫码解锁任何**付费**内容（无内购规避）；③ 二维码是给「被提醒人」扫，不是 app 用户的唯一登录方式（否则可能触发 4.8 需提供 Sign in with Apple）；④ App Review Notes 里说明「二维码用于让被提醒人关注公众号接收通知」，审核员（海外、无微信）才能理解；⑤ 隐私政策披露收集 openid/昵称。

## 附录 B：备选方案——小程序订阅消息

若拿不到企业主体的认证服务号：改用微信小程序的「订阅消息」。差异：① 小程序个人主体也可申请；② 每条提醒需用户**当次点击授权**（一次性订阅），不能长期静默推送；③ 需做一个极简小程序承载授权与落地页。体验弱于服务号，但门槛低。后端发送接口换成 `message/subscribe/send`，其余结构不变。

## 附录 C：关键约束速记

- 必须**服务号**且**已微信认证**；订阅号/个人号不行。
- 回调与接口强制 **https + 已备案域名**。
- access_token 有频率与 IP 白名单限制；本后端已做内存缓存。
- 模板消息内容受模板审核约束，不可随意拼接长文本。
- 阿里云安全组只放行 22/80/443/18088，后端走内网 9528 + nginx 443 反代。
