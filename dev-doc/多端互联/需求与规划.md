# 侧边待办 · 多端互联 需求与规划

> 状态：**规划阶段**（仅设计，未开发）
> 本期落地范围：**仅 mac 端** + 云端账号 + 每分钟轮询同步
> 企微、移动端、Web 端：列入路线图，本期不做

---

## 一、一句话目标

让同一个用户在多个端（本期 mac，后期 iOS/Web）共享同一份待办；并能接收「别人指派」「自己其他端同步」的待办，本人批准后进入正式列表。

---

## 二、核心概念

### 2.1 云端唯一 ID
- 用户**扫一个二维码绑定个人微信**，云端取微信 **unionid** 作为账号唯一标识。
- **昵称**首次取微信昵称，之后用户可在 app 内**自行编辑**（与微信解耦）。
- 一个唯一 ID 在所有端共享同一份数据。
- 不使用 AppleID（无法跨平台），故必须有云端账号。

> ⚠️ 待你确认的前置条件：
> 1. 要拿到稳定的 **unionid**，你的「服务号」与未来的 app 必须挂在**同一个微信开放平台账号**下（unionid 是开放平台级唯一，openid 只在单个公众号/应用内唯一）。
> 2. 微信扫码回调要求 **HTTPS + 已备案域名**，需要一台云服务器。

### 2.2 待办来源（每条待办带来源标记）
| 来源 | 说明 | 是否需批准 |
|---|---|---|
| `self` | 本端创建 | 否 |
| `synced` | 自己其他端同步过来 | 否 |
| `assigned` | 别人指派（团队版） | **是** |
| `wecom`（后期） | 企微自建应用回调推送 | 是 |

## 三、个人端需求（本期 mac）

1. **扫码绑定页**：未登录时展示二维码，扫码绑定微信后进入主界面。
2. **收件箱 / 待批准分区**：`assigned` / `wecom` 来源的待办先进这里，
   - 批准 → 进正式待办列表，回执给指派人；
   - 拒绝 → 丢弃，回执给指派人（带可选理由）。
3. **每分钟轮询**：客户端定时拉取「属于我的」增量数据（别人指派的、其他端同步的、远端删除的）。
4. **同步状态指示**：顶部小圆点显示「同步中 / 上次同步时间」。
5. **账号页**：微信头像、昵称（可编辑）、唯一 ID（unionid，脱敏展示）、解绑、退出。

---

## 四、团队端需求（团队版分支）

1. **唯一 ID 接收他人指派的待办，本人批准后才进列表**（同 3.2）。
2. **@指派**：
   - 在待办输入框输入 `@` → 弹出**人员选择器**；
   - 选择器**可搜索**（按昵称/拼音）；
   - 选择器**可多选**（一次指派给多人）；
   - 选中以 chip 形式回填输入框。
3. **团队成员清单**：同一团队/组织内的唯一 ID 列表（昵称 + 头像 + 在线/最近同步），是 @选择器的数据源。

## 五、数据模型（云端，草案）

```
users
  uid            主键 = 微信 unionid
  nickname       用户可编辑昵称
  avatar_url     微信头像
  wx_openid      服务号内 openid（回调用）
  created_at

todos
  id             待办主键（UUID）
  owner_uid      归属人（= 谁的列表）
  text, deadline, color
  source         self | synced | assigned | wecom
  assigner_uid   指派人（source=assigned 时）
  approval       pending | approved | rejected
  updated_at, deleted

teams
  team_id, name
  member_uids[]  成员唯一 ID 列表
```

> 多租户隔离铁律：所有查询必须带 `owner_uid` 过滤，绝不让一个 uid 读到别人的待办。

---

## 六、同步协议（每分钟轮询）

- 客户端持 `last_cursor`（服务端版本号/时间戳）。
- **下行**：`GET /api/sync?uid=X&since=<cursor>`
  返回 `{ todos:[增量], pending:[待批准], deleted:[id...], server_time, next_cursor }`
- **上行**：本端创建/批准/完成/删除 → `POST /api/todos`（带 uid 鉴权 token）。
- **合并规则**：`approved` 进列表，`pending` 进收件箱；冲突用 `updated_at` 后写赢（LWW），简单够用。
- 轮询沿用现有 AI 帮手定时器的节奏（每分钟一批）。

## 八、关键决策点（需你拍板）

1. 是否已有微信开放平台账号？服务号能否与之绑定（决定能否拿 unionid）。
2. 云服务器 + 备案域名 + HTTPS 的准备（微信回调强制要求）。
3. 「团队」的组织边界怎么定义：手动建团队拉人，还是按企微部门？（影响成员清单来源）
4. 待办上行鉴权方式：扫码后下发的长期 token，还是每次带签名。
