## 项目概述

便笺 (Sticky) — 轻量级侧边栏待办 + 纪念日管理工具。纯前端，无后端，数据存储在 localStorage。

## 技术栈

| 层 | 技术 | 说明 |
|---|---|---|
| 前端 | HTML/CSS/JS | 原生，无框架，单页应用 |
| 存储 | localStorage | 待办、纪念日、设置持久化 |
| 设计 | oklch 色彩系统 | 5 主题色、毛玻璃面板 |

## 关键目录

| 目录 | 说明 |
|---|---|
| `frontend/` | 前端代码 |
| `backend/` 或 `server/` | 后端代码 |
| `docs/` | 产品文档 |

## 三铁律（执行者协作协议）

### 1. 做前只问：有没有活干
- 读待办事项文件，顶部任务队列 → 抢 #1
- `[ ]` 默认都可以做，除非旁边写 `⏸ 阻塞：{原因}`
- 遇到歧义 → 追加 `❓ 问规划者：{一行问题}`，不要自己选方案

### 2. 做完可以自己打 `[x]`
- **硬条件**：commit 存在 + `dev-doc/` 三文档齐全且非空
- 条目末尾追加：`✅ 完成 @ {yyyy-MM-dd HH:mm:ss} by commit {hash}`

### 3. 新增任务追加顶部，完成后下沉到「最近完成」
- 必须以待办文件中的状态为准，不要自己维护缓存

## 编译错误处理规则

- 遇到编译/构建错误时，**自行修复，不询问用户**
- 最多尝试 3 轮自动修复
- 3 轮后仍失败，说明阻塞点和建议，等待用户决策

## 文件和文件夹命名规范

- 禁止花括号 `{}`、特殊字符、空格
- 使用 kebab-case 或 snake_case
- 文件名必须能反映内容

## 代码生成规范

- **禁止硬编码数据** — 所有方法必须参数化
- **禁止使用 localhost / 127.0.0.1** — 从配置读取
- **测试数据通过脚本管理** — 不在代码中写假数据

## 任务完成标准

一个任务只有满足以下全部条件才算完成：
1. 代码已提交（commit 存在）
2. dev-doc 三文档齐全
3. 构建通过（类型检查 + 生产构建）
4. curl / 浏览器验证通过

## Docker 使用规范

- 所有中间件用 Docker 启动
- 不修改已有容器
- 端口映射用冷门端口

## 工程约束（速查表）

| 类别 | 约束 |
|---|---|
| **时间格式** | `yyyy-MM-dd HH:mm:ss`，不带 T，不换行 |
| **敏感凭证** | 放 `.env` 或 `.secrets/`（gitignored），禁止硬编码 |
| **commit 前检查** | `git status` 确认没带上 `.env` / `.secrets/` |
| **品牌名** | 严格统一写法，禁止变体 |
| **用户可见文案** | 禁止中英夹杂（技术术语白名单例外） |
| **端口选择** | 用冷门端口（9527/9528 等），不用 3000/8080 默认端口 |
| **构建验证** | 类型检查 + 生产构建必须双通过 |
| **敏感操作** | 回滚、重置、删除前先备份 |
| **SSH 连接** | 严禁 `sshpass -p`，必须用 `sshpass -f <文件>` 或 SSH 密钥 |

## 自动行为（每次对话必须执行）

1. **写代码前先读案例库**，对照历史踩坑避免重复犯错
2. **接到任务先分类**（需求设计/Bug修复/部署），走对应工作流程
3. **完成功能后立即 commit + dev-doc 三文档**，不要攒多个功能
4. **修复 bug → 立即追加到案例库**（格式：场景/现象/原因/修复/**教训加粗**）
5. **验证失败 → 自动修复最多 3 轮**，3 轮后报告阻塞点
